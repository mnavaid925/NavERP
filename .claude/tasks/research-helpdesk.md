## Feature catalog by sub-module
### 3.36 Helpdesk

- **Ticket raise/track/resolve lifecycle** — a status state-machine (new → open/in-progress → waiting-on-employee → resolved → closed) an agent walks a ticket through · seen in: Zendesk, Freshservice, Zoho Desk, HappyFox, Zoho People · priority: table-stakes · spine: new table `HelpdeskTicket` (requester reuses `hrm.EmployeeProfile`) · buildable now
- **Ticket priority (Low/Medium/High/Critical or Urgent)** — drives SLA targets and agent triage order · seen in: Zendesk, Freshservice, Freshdesk, Zoho Desk · priority: table-stakes · spine: field on `HelpdeskTicket` · buildable now
- **Requester / raised-by identity** — every ticket is tied to the employee who opened it · seen in: all 10 · priority: table-stakes · spine: reuse `hrm.EmployeeProfile` (never a new employee table) · buildable now
- **Resolution note / closing summary** — the agent's final answer recorded on the ticket when solved · seen in: Zendesk (Solved comment), SolarWinds ("description and resolution become a solution") · priority: table-stakes · spine: field on `HelpdeskTicket` (`resolution_notes`) · buildable now
- **Multi-turn conversation thread (public replies + internal notes)** — a running back-and-forth on the ticket · seen in: Zendesk, Freshdesk, Freshservice, Jira Service Management · priority: common · spine: new child table (mirrors `crm.CaseComment`) · buildable now, **deferred this pass** (see below) — one `resolution_notes` field covers "track/resolve" for a lean v1
- **Ticket origin/channel (portal, email, phone, chat, walk-in)** — records how the ticket arrived · seen in: Zendesk, HappyFox, ServiceNow (Employee Center + chatbot) · priority: common · spine: field on `HelpdeskTicket`, mirrors `crm.Case.origin` · buildable now (field only — actual email/chat ingestion is integration/later)
- **Ticket categories (admin-managed taxonomy, e.g. HR/IT/Admin/Facilities)** — a configurable lookup employees pick when raising a ticket · seen in: Zoho People, HappyFox, SolarWinds, Spiceworks, Jira SM (request types) · priority: table-stakes · spine: new table `HelpdeskCategory` (tenant-configurable, mirrors `crm.KbCategory`) · buildable now
- **Category → default agent / agent pool routing** — each category has an assigned agent (or team) so tickets auto-route on create · seen in: Zoho People ("assign HR agents to each category"), ServiceNow (routes "based on category, urgency, business rules") · priority: differentiator · spine: `HelpdeskCategory.default_assignee → auth User`; simple auto-assign-on-create is buildable now — full load-balanced/round-robin routing (Freshdesk Omniroute) is integration/later
- **Category → department routing** — ties a category to the owning org unit (e.g. "Facilities" → Facilities department) · seen in: ServiceNow (routes to "appropriate support group"), Jira SM (queues per team) · priority: common · spine: `HelpdeskCategory.department → core.OrgUnit(kind="department")` (reuse, mirrors `Announcement` audience targeting) · buildable now
- **Agent assignment (manual reassign)** — an HR/IT/Admin agent is assigned/reassigned to work a ticket · seen in: all 10 · priority: table-stakes · spine: `HelpdeskTicket.assignee → auth User` · buildable now
- **Automated round-robin / load-based / skill-based ticket assignment** — the system picks the next agent by workload/skill, not a human · seen in: Freshdesk (Omniroute), Freshservice, ServiceNow · priority: differentiator · spine: would need an agent-capacity/skill table · integration/later — out of scope for a lean pass, category-default-agent covers the common case
- **SLA policy: per-priority first-response + resolution time targets** — a named policy defines hours-to-first-response and hours-to-resolution for each priority level · seen in: Freshservice, Zendesk, Zoho Desk, HappyFox, Jira SM, SolarWinds, Zoho People · priority: table-stakes · spine: new table `HelpdeskSLAPolicy` (mirrors `crm.SlaPolicy` field-for-field: `response_low/medium/high/critical`, `resolution_low/medium/high/critical` in hours, `targets_for(priority)`) · buildable now
- **SLA due-date computation + breach flag** — `first_response_due`/`resolution_due` computed from the policy on ticket creation; `is_overdue`/`is_response_overdue` flags for an open, past-due ticket · seen in: all SLA-capable products above · priority: table-stakes · spine: fields + properties on `HelpdeskTicket` (mirrors `crm.Case.save()` policy-driven due-date logic exactly) · buildable now
- **Per-category SLA override** — a category can carry its own default SLA policy distinct from the tenant default · seen in: HappyFox ("unique SLA for each ticket category"), SolarWinds (SLA scoped by category/subcategory), Zoho People (SLA set per category) · priority: common · spine: `HelpdeskCategory.default_sla_policy → HelpdeskSLAPolicy` · buildable now
- **Multi-level SLA-breach escalation (notify manager/next tier)** — automatic escalation chain when a ticket nears/misses SLA · seen in: Freshservice (up to 4 escalation levels), HappyFox (auto-escalate + reassign) · priority: differentiator · spine: none new (would ride on the existing overdue flags) · integration/later (needs a scheduled job + notification channel)
- **Business-hours-aware SLA clocks (pause outside working hours/on hold)** — SLA timers stop on weekends/holidays or while "on hold" · seen in: Freshservice, Jira SM, SolarWinds · priority: differentiator · spine: none · integration/later — `crm.SlaPolicy`'s own docstring already defers this ("Business-hours calendars are deferred"); Helpdesk inherits the same deferral for consistency
- **Knowledge base / self-help articles (FAQs)** — a searchable library of how-to/troubleshooting articles employees browse before raising a ticket · seen in: Freshservice, Zendesk, Zoho Desk, HappyFox, ServiceNow, SolarWinds, Spiceworks, Zoho People · priority: table-stakes · spine: new table `KnowledgeArticle` (mirrors `crm.KnowledgeArticle`, trimmed) · buildable now
- **KB categorization** — articles are organized under the same category taxonomy as tickets (Zoho People explicitly attaches FAQs to each case category) · seen in: Zoho People, ServiceNow, Zendesk · priority: common · spine: `KnowledgeArticle.category → HelpdeskCategory` (**reuse the ticket-category table for both** — avoids a second taxonomy table) · buildable now
- **Draft/Published/Archived article lifecycle** — articles are authored, reviewed, then published or retired · seen in: Zendesk, Zoho Desk, ServiceNow · priority: table-stakes · spine: `status` field on `KnowledgeArticle` (mirrors `crm.KnowledgeArticle.STATUS_CHOICES`) · buildable now
- **AI-assisted article generation / auto-triage / virtual agent chatbot** — AI drafts KB content or auto-categorizes/answers tickets · seen in: Freshservice (AI-driven KB), ServiceNow (virtual agent), Zendesk (AI triage) · priority: differentiator · spine: none · integration/later (external LLM)
- **Article view counters / helpful-not-helpful voting / public portal token** — self-service usage analytics and an unauthenticated public article link · seen in: Freshservice ("usage insights"), `crm.KnowledgeArticle` (existing NavERP precedent: `views_count`/`helpful_count`/`public_token`) · priority: common · spine: would extend `KnowledgeArticle` · buildable now, **deferred this pass** — HRM's KB is internal-only (employees are already authenticated users), so no public portal token is needed the way CRM needs one for external customers
- **"Resolved ticket becomes a KB solution"** — closing a ticket can spin off its description+resolution as a new KB article · seen in: SolarWinds ("description and resolution become a solution") · priority: differentiator · spine: a manual "create article from ticket" action (pre-fills a `KnowledgeArticle` from the ticket) · buildable now (as a small convenience view, not a required field)
- **Post-resolution satisfaction survey (CSAT)** — a 1–5 (or thumbs/emoji) rating + optional comment collected once a ticket is resolved/closed · seen in: Zendesk, Freshdesk, Zoho Desk, Zoho People · priority: table-stakes · spine: fields on `HelpdeskTicket` (`satisfaction_rating`/`satisfaction_comment`/`satisfaction_at` — mirrors `crm.Case`'s inline CSAT fields exactly; **no separate survey model needed**, matching the existing NavERP precedent) · buildable now
- **CSAT trend analytics / dashboard rollups (avg CSAT, escalation counts, resolution-time trend)** — reporting on the collected satisfaction/SLA data · seen in: Zoho People, Freshdesk, Freshservice · priority: common · spine: computed off `HelpdeskTicket` (no new table) · buildable now, **deferred this pass** — a dashboard/report view is a fast follow, not required to ship the 4 core models
- **Self-service employee portal / Employee Center** — a branded portal where employees raise tickets and browse the KB without seeing agent tooling · seen in: ServiceNow (Employee Center), Zendesk, HappyFox · priority: common · spine: none new — unlike `crm.CustomerPortalAccess` (built for *external* customers who have no ERP login), HRM employees are already authenticated tenant users via `hrm.EmployeeProfile`/`auth.User`, so ticket-raising is just a normal logged-in view, no portal-access binding table required · buildable now

## Recommended build scope (this pass — 4 models)
- **`HelpdeskCategory`** [TenantOwned, no NUMBER_PREFIX — a simple admin catalog like `LeaveType`/`JobGrade`] — `name`, `description`, `department → core.OrgUnit(kind="department")` (nullable, reuse), `default_assignee → auth.User` (nullable), `default_sla_policy → HelpdeskSLAPolicy` (nullable, FK forward-declared/string ref), `is_active`. Justified by: *Ticket Categories* (NavERP.md bullet), *category→agent routing* (Zoho People), *category→department routing* (ServiceNow), *per-category SLA override* (HappyFox/SolarWinds/Zoho People). Seeded with HR/IT/Admin/Facilities to match the NavERP.md bullet text. Doubles as the KB taxonomy (no separate KB-category table).
- **`HelpdeskSLAPolicy`** [TenantNumbered, `NUMBER_PREFIX = "HSLA"`] — `name`, `description`, `is_active`, `is_default`, `response_low/medium/high/critical` (hours), `resolution_low/medium/high/critical` (hours), `targets_for(priority)`. Justified by: *SLA Management* (NavERP.md bullet). Field-for-field mirror of `crm.SlaPolicy` — same per-priority hour targets, same `targets_for()` contract, same "business-hours calendars deferred" caveat.
- **`HelpdeskTicket`** [TenantNumbered, `NUMBER_PREFIX = "TKT"`] — `requester → hrm.EmployeeProfile` (CASCADE, the anchor), `category → HelpdeskCategory` (SET_NULL, nullable), `subject`, `description`, `priority` (low/medium/high/critical), `status` (new/open/in_progress/waiting/resolved/closed + `OPEN_STATUSES`), `origin` (portal/email/phone/chat/walk_in), `assignee → auth.User` (SET_NULL, nullable — auto-defaulted from `category.default_assignee` on create), `sla_policy → HelpdeskSLAPolicy` (SET_NULL, nullable — auto-defaulted from `category.default_sla_policy` or the tenant's `is_default` policy), `first_response_due`/`first_responded_at`/`resolution_due`/`resolved_at`/`closed_at` (all system-set in `save()`, mirroring `crm.Case.save()`'s SLA/timestamp logic verbatim), `resolution_notes` (agent's closing summary), `satisfaction_rating`/`satisfaction_comment`/`satisfaction_at` (CSAT, system-set via a dedicated feedback action, only when resolved/closed). Justified by: *Ticket Management*, *SLA Management* (consumer side), *Satisfaction Survey* — all three NavERP.md bullets converge on this one model, exactly mirroring how `crm.Case` carries its own inline SLA + CSAT fields rather than spawning satellite tables.
- **`KnowledgeArticle`** [TenantNumbered, `NUMBER_PREFIX = "KBA"`] — `title`, `category → HelpdeskCategory` (SET_NULL, nullable, reuses the ticket taxonomy), `body`, `status` (draft/published/archived), `owner → auth.User` (SET_NULL, nullable). Justified by: *Knowledge Base* (NavERP.md bullet). Trimmed from `crm.KnowledgeArticle` — no `public_token`/view-counters/visibility split, since this KB is internal-only (employees are already logged-in tenant users, no external "customer" audience to gate).

**Ticket status-action views** should NOT reuse the `_hr_request_*` draft→pending→approved/rejected helpers verbatim — those model a single-approver *authorization* workflow (DocumentRequest/IdCardRequest/AssetRequest/Suggestion/TravelRequest all end in one person approving or rejecting). A helpdesk ticket is an *agent-worked* item with a different shape (assign → start → resolve → close → reopen, no approve/reject). Build bespoke `ticket_assign`/`ticket_start`/`ticket_resolve`/`ticket_close`/`ticket_reopen`/`ticket_feedback` actions instead, following `crm.Case`'s status-transition + inline-SLA/CSAT pattern (same app family, same shape) rather than the HRM request-approval family.

## Deferred (later passes / integrations)
- **`TicketComment`** (multi-turn public reply + internal-note conversation thread, mirrors `crm.CaseComment`) — deferred; `resolution_notes` covers "track/resolve" for a lean v1; add this table if/when the module needs a full agent↔employee back-and-forth.
- **Automated round-robin / load-based / skill-based ticket routing** (Freshdesk Omniroute-style) — needs an agent-capacity/skill model; deferred, `HelpdeskCategory.default_assignee` covers simple category-based auto-assignment for now.
- **Multi-level SLA-breach escalation notifications** (Freshservice's up-to-4-tier escalation) — needs a scheduled job + notification channel; the `is_response_overdue`/`is_resolution_overdue` flags ship now, the notify/escalate action is a follow-up.
- **Business-hours-aware SLA clocks** (pause outside working hours/holidays) — explicitly deferred in `crm.SlaPolicy` already; Helpdesk inherits the same deferral (24×7 calendar-hour math only).
- **KB view counters, helpful/not-helpful voting, public unauthenticated article token** — `crm.KnowledgeArticle` precedent exists for these; skipped here because HRM's KB is internal-only (no external audience needing a public link).
- **CSAT/SLA analytics dashboard** (avg CSAT trend, escalation counts, resolution-time chart) — computed off `HelpdeskTicket`, no new table; a fast-follow reporting view, not required to ship the 4 models.
- **"Create KB article from resolved ticket" convenience action** — nice-to-have (SolarWinds pattern); small view-layer addition, not a schema requirement.
- **AI-assisted triage / auto-categorization / virtual-agent chatbot / AI-generated KB drafts** — external LLM integration (Freshservice, ServiceNow, Zendesk); out of a single Django pass.
- **Self-service Employee Center portal UI polish** (dedicated branded portal distinct from the agent workspace) — the raise-ticket/browse-KB views ship as normal authenticated employee-facing pages this pass; a fully separate "portal" chrome is cosmetic and can follow later.
